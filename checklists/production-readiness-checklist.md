# 生产就绪检查清单

> 适用场景：服务首次上生产 / 重大版本发布
> 前置技能：完整走过 `brainstorming` → `writing-plans` → `TDD` → `verification-before-completion`

---

## 🚨 阻塞型检查

### 配置管理

- [ ] 生产配置**独立**于开发 / staging（不同 etcd key、不同 DB 实例）
- [ ] 所有 secret 走**环境变量**或 secret manager（Vault / KMS），**不在** yaml 明文
- [ ] 配置变更有版本控制和审计
- [ ] Feature flags 就位，可灰度开关

### 资源与容量

- [ ] 已做**压测**并记录基线（QPS、P99 延迟、错误率）
- [ ] 已评估峰值容量（业务峰值 × 3 倍余量）
- [ ] Pod / 容器资源 requests 和 limits 设置合理
- [ ] HPA 策略配置（CPU / 内存 / 自定义指标）
- [ ] 数据库连接池大小合理（单实例连接数 × 实例数 ≤ DB 最大连接数的 70%）

### Resilience（韧性）

- [ ] **超时**：所有外部调用有超时（DB、Redis、RPC、HTTP client）
  - 验证：`grep -rn "Timeout" etc/*.yaml`
- [ ] **熔断**：go-zero breaker 默认启用（无需手动配置，但验证未被关闭）
- [ ] **限流**：入口限流（`periodlimit` / `tokenlimit`）
- [ ] **降级**：关键接口在依赖故障时有降级策略
- [ ] **重试**：只对幂等操作重试，带指数退避
- [ ] **负载削峰**：启用 `load.Shedder` 或类似机制

### 可观测性（三支柱）

**Metrics**
- [ ] Prometheus 指标暴露（`/metrics` endpoint）
- [ ] 关键业务指标有 dashboard
- [ ] RED 指标（Rate / Errors / Duration）可查
- [ ] 告警规则配置（错误率 > 阈值、P99 延迟异常）

**Logs**
- [ ] 结构化日志（`logx.WithContext` + JSON 格式）
- [ ] 日志级别合理（生产不开 DEBUG）
- [ ] 日志聚合到 ELK / Loki / 云服务
- [ ] 日志不包含敏感信息（密码、token、身份证）
- [ ] 日志采样策略合理（高频日志采样）

**Tracing**
- [ ] 链路追踪启用（Jaeger / SkyWalking）
- [ ] 跨服务 ctx 透传正确
- [ ] 关键 span 包含业务属性

### 部署与运维

- [ ] **健康检查**：
  - [ ] `/healthz` liveness endpoint 响应快（< 100ms）
  - [ ] `/readyz` readiness 真实检查依赖（DB、Redis、RPC）
- [ ] **优雅关闭**：`SIGTERM` 后完成 in-flight 请求再退出
- [ ] **部署策略**：滚动更新 / 蓝绿 / 金丝雀（至少一种）
- [ ] **回滚方案**：一键回滚到上个版本
- [ ] **数据库迁移**：已按 `db-migration-checklist.md` 执行
- [ ] **依赖服务**：上下游依赖已就位（包括第三方服务）

### 安全

- [ ] **TLS**：对外接口强制 HTTPS
- [ ] **内部 RPC**：考虑 mTLS（或在 VPC 内安全域）
- [ ] **认证**：所有非公开接口有 JWT / OAuth 鉴权
- [ ] **授权**：RBAC / ACL 业务权限校验
- [ ] **输入校验**：所有入参有 validate tag
- [ ] **SQL 注入**：参数化查询（go-zero sqlx 默认防护）
- [ ] **速率限制**：防爆破 / 防刷
- [ ] **审计日志**：敏感操作（登录、支付、权限变更）有审计
- [ ] **依赖扫描**：`go list -json -m all | nancy sleuth` 或 govulncheck
- [ ] **镜像扫描**：Trivy / Grype 扫描容器镜像

### 合规与数据

- [ ] 敏感数据字段加密存储（身份证、手机、密码哈希）
- [ ] GDPR / 数据本地化要求已满足（如适用）
- [ ] 数据备份策略已配置（频率、保留期、恢复测试）
- [ ] 日志保留符合合规要求

### 文档

- [ ] **Runbook**：常见故障处理手册
- [ ] **Oncall 文档**：值班响应流程、联系人
- [ ] **架构图**：部署拓扑、数据流、依赖关系
- [ ] **API 文档**：对调用方公开的接口文档
- [ ] **CHANGELOG**：本次发布内容

---

## ⚠️ 建议型检查

### 成本

- [ ] 资源使用效率评估（CPU / 内存是否过度分配）
- [ ] 云服务成本估算（存储、流量、API 调用）
- [ ] 日志量可控（避免日志费用失控）

### 灾备

- [ ] 多可用区部署
- [ ] 数据库主从 / 多活
- [ ] 跨地域容灾（RTO / RPO 符合 SLA）
- [ ] 定期灾备演练

### 混沌工程

- [ ] 依赖故障场景已测试（DB down、Redis down、RPC timeout）
- [ ] 网络分区场景已测试
- [ ] 节点 kill 场景已测试

## 发布前最终确认

**在点击「发布」之前，回答以下问题：**

1. **「如果现在出故障，我能在 5 分钟内定位到哪一层吗？」** → 可观测性检查
2. **「如果要回滚，我 10 分钟内能做到吗？」** → 回滚方案检查
3. **「如果流量突增 10 倍，服务会崩吗？」** → 容量与限流检查
4. **「如果依赖挂了，服务会完全不可用还是降级？」** → Resilience 检查
5. **「数据出问题了，能从备份恢复吗？」** → 灾备检查

**任何一个答案是「不知道」或「可能不行」→ 不要发布。**

# 数据库迁移检查清单

> 适用场景：数据库 schema 变更（新表、新字段、索引、类型变更）
> 前置技能：`superpowers:brainstorming`（必须讨论兼容性）

---

## 🚨 阻塞型检查

### 迁移脚本本身

- [ ] 迁移文件命名符合规范（时间戳前缀 + 描述）
  - 例：`20260411_120000_add_user_email_verified.up.sql`
- [ ] 提供 `up` 和 `down` 两个方向
- [ ] `down` 脚本可安全回滚（已测试）
- [ ] SQL 使用目标数据库方言（MySQL / PostgreSQL 语法不混用）
- [ ] 脚本**幂等**（重复执行不出错）或迁移工具保证只执行一次

### 向后兼容性（关键）

- [ ] **新增字段**：
  - [ ] 新字段**允许 NULL** 或有**默认值**（避免已有行插入失败）
  - [ ] 应用代码已先发布能写入新字段的版本
- [ ] **删除字段**：
  - [ ] 应用代码已先发布**不再读写**该字段的版本
  - [ ] 延迟到下一次发布再真正删除
- [ ] **字段重命名**：**禁止直接改名**，应分三步：
  1. 新增目标字段
  2. 应用双写并迁移数据
  3. 下个版本删除旧字段
- [ ] **字段类型变更**：
  - [ ] 确认新类型能容纳所有现有数据（`INT → BIGINT` 可以，反之不行）
  - [ ] 字符串变短需先 check `MAX(LENGTH(col))`
- [ ] **NOT NULL 约束**：
  - [ ] 先允许 NULL → 回填数据 → 再加约束（分多次迁移）

### 索引变更

- [ ] **新增索引**：
  - [ ] 大表使用 `CREATE INDEX CONCURRENTLY`（PostgreSQL）或 `ONLINE`（MySQL 5.6+）
  - [ ] 评估索引大小和写入性能影响
- [ ] **删除索引**：
  - [ ] 确认没有查询依赖该索引（`EXPLAIN` 检查常用 query）
- [ ] **唯一索引**：
  - [ ] 已确认现有数据无重复
  - [ ] 有处理冲突的策略

### 锁与停机

- [ ] 评估迁移对表的锁定时间
- [ ] 大表变更使用在线 DDL 工具（`pt-online-schema-change` / `gh-ost`）
- [ ] 高峰期**禁止**执行阻塞型迁移
- [ ] 预估迁移耗时并告知运维

### 数据迁移

- [ ] 大批量数据迁移**分批**执行（每批 1000-10000 行）
- [ ] 迁移脚本有进度日志和断点续传
- [ ] 迁移期间应用行为明确定义（双写 / 只读 / 维护页）

### 测试

- [ ] 在开发环境 + staging 环境完整跑过一遍 up 和 down
- [ ] 单元测试更新 model 层（`goctl model mysql datasource` 重新生成）
- [ ] 集成测试在干净数据库 + 有数据的数据库都能通过
- [ ] 性能回归测试（迁移后查询性能无明显下降）

---

## ⚠️ 建议型检查

### 监控与回滚

- [ ] 迁移前已备份（或确认备份最近的 PITR 可用）
- [ ] 定义回滚决策点（"迁移 N 秒内未完成则回滚"）
- [ ] 迁移后监控关键指标（错误率、延迟、连接数）
- [ ] 团队已就位，可响应异常

### 文档

- [ ] Schema 变更在 CHANGELOG 记录
- [ ] 受影响的接口文档已更新
- [ ] Runbook 包含此次迁移的操作步骤

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| `ALTER TABLE ADD COLUMN NOT NULL DEFAULT 'x'` 大表阻塞 | 分两步：先加字段（允许 NULL）→ 回填 → 加约束 |
| 直接 `DROP COLUMN` 导致旧版本应用崩溃 | 先发布"不再读写"的代码，再删除字段 |
| `RENAME COLUMN` 导致在途流量失败 | 双写三步走：加新列 → 双写 → 删旧列 |
| 新增唯一索引时存在重复数据 | 先清理重复 → 再加索引 |
| `COUNT(*)` 慢导致迁移脚本超时 | 用 `COUNT(id)` 或从 `information_schema` 查估值 |
| 迁移脚本漏 `down` | 始终成对提供，哪怕 `down` 是 no-op 也要注明 |
